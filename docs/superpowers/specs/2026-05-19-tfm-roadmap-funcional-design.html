TFM RAG Chatbot Platform — Diseño funcional + roadmap

Autor: Ade Cabo Cuartero · Fuente: brainstorming sesiones 1-4 (2026-05-19 / 2026-05-20) · Estado: 15/15 secciones cerradas · Próximo paso: invocar writing-plans tras revisión del usuario y luego extraer OpenSpec proposals (1 CAP → 1 proposal).

§1Visión, glosario y supuestos

Visión

Plataforma open-source, autoalojable, multi-tenant que permite a un usuario no técnico crear y desplegar chatbots RAG de atención al cliente conectados a sus propios documentos y bases de datos, sin escribir código. Cada chatbot se publica como un widget embebible en la web del cliente final. El sistema es agnóstico al proveedor de LLM/embeddings: catálogo de adaptadores con Ollama local por defecto + endpoints OpenAI-compatibles para proveedores cloud (OpenAI, Groq, Together, OpenRouter, DeepSeek, …).

Usuarios

  • Admin del tenant (en MVP = único usuario por tenant): crea KnowledgeBases, configura chatbots, integra credenciales de proveedores, supervisa conversaciones y obtiene el snippet del widget.
  • Usuario final del widget: anónimo, chatea con un chatbot embebido en la web del cliente.
  • Investigador (TFM): ejecuta evaluación RAGAS por CLI sobre escenarios controlados para validar la calidad del sistema.

Supuestos clave del MVP

  1. Multi-tenant real (1 admin = 1 tenant); sin invitaciones de miembros (post-MVP).
  2. Autenticación: email+password obligatorio + Google OAuth opcional. JWT 24h sin refresh tokens (re-login al expirar).
  3. El catálogo de proveedores vive en código: no hay superadmin.
  4. La ingesta es asíncrona con FastAPI BackgroundTasks + tabla ingestion_jobs + polling. Sin Celery/Redis en MVP.
  5. Vector store: siempre Qdrant. Una colección física por (tenant, dim) — permite múltiples embeddings por tenant.
  6. Ollama corre local junto al backend (servicio en docker-compose.yml); cada tenant nuevo arranca con un ProviderCredential Ollama default.
  7. Streaming parcial: las iteraciones del loop agéntico se emiten en vivo vía SSE (tool en curso, estado de cada step) para que el usuario vea el "pensamiento" del agente. La respuesta final puede entregarse completa o en streaming según el endpoint.
  8. Idioma de la plataforma: español para UI/spec/log; código y nombres de clase en inglés.
  9. Stack frontend: Next.js (App Router) + NextAuth + shadcn/ui (alineado con la memoria del TFM).
  10. Deploy del MVP: solo local con docker-compose up (tribunal corre el stack).
  11. Licencia: AGPLv3 (del PDF del TFM).

§3Flujos end-to-end

Ocho flujos cubren el journey de 21 pasos. Los flujos B y C son ahora agentic loops en lugar de pipelines lineales. F y G pertenecen a una KnowledgeBase, no a un chatbot. H es nuevo: diagrama de secuencia del loop agéntico.

Reescritos respecto a la primera versión: B y C ahora son agentic loops; F y G pertenecen a una KB (no a un chatbot); flujo H nuevo.

A — Ingesta de archivos locales

Cliente panel  →  POST /api/knowledge-bases/{kb}/sources/documents  (multipart)
              →  persistir DocumentSource (status=not_started)
              ←  201 DocumentSource

Cliente panel  →  POST /api/sources/{id}/ingest
              →  crear IngestionJob (status=queued)
              →  FastAPI BackgroundTasks dispara worker
                    worker: DocumentLoader.load() (segun extensión)
                          → Chunker.chunk(ChunkingConfig de la KB)
                          → EmbeddingProvider.embed_batch(EmbeddingSelection de la KB)
                          → VectorStore.upsert(payload {tenant_id, kb_id, source_id, ...})
                          → IngestionJob.status = done
              ←  202 IngestionJob

Cliente panel  →  GET /api/ingestion-jobs/{job_id}  (polling 2s)
              ←  200 IngestionJob (status: queued | running | done | failed)

B — Chat documental (agentic, vía playground o widget)

Cliente  →  POST /api/chatbots/{id}/chat  (o /api/public/...)
        →  AnswerQuery.execute(chatbot_id, session_id?, message)
              loop (hasta max_retrieval_iterations):
                  Router.route(state, available_tools)  → ToolCall
                  if tool == search_docs:
                      RetrieveDocs(kb_ids, query, top_k, threshold)
                      → embed + Qdrant filtro + opcional Reranker
                  elif tool == query_database:
                      ExecuteSQL(database_source_id, nl_query)
                      → text2SQL + 3 guards + execute read-only
                  elif tool == final_answer:
                      LLMProvider.generate(context acumulado)
                      break
                  elif tool == abstain:
                      respuesta honesta de "no puedo"
                      break
              persistir ChatMessage con metadata.iterations[]
        ←  200 { session_id, message }

C — Chat con docs + SQL (caso compuesto)

Es el mismo flujo B; el agentic loop decide en cada turno entre search_docs y query_database. No hay pipeline lineal: el LLM puede alternar tools, hacer una búsqueda, ver el resultado, decidir hacer otra, etc., hasta el tope.

D — Login/registro

email+pw:
  Cliente  →  POST /api/auth/register {email,password}
          →  RegisterUser.execute → BootstrapTenant.execute
                                    (crea Tenant
                                     + ProviderCredential Ollama default
                                       (config_source=SERVER_ENV))

              →  GoogleOAuthVerifier valida (firma, aud, iss, exp)
              →  busca user por email; si no existe → RegisterUser + BootstrapTenant

E — Crear chatbot (wizard single-POST)

Wizard cliente (7 pasos):  Identidad → Cerebro → Conocimiento → Pipeline
                            → Widget → Revisión → Crear
  Estado se mantiene en cliente; NO hay draft en BD.

Cliente  →  POST /api/chatbots {name, system_prompt, llm_selection,
                                 kb_ids[], pipeline_config, widget_config}
        →  validar embedding compatibility cross-KB
              si KBs[].embedding_selection no son iguales → 409 IncompatibleEmbeddingsError
        →  persistir Chatbot + tabla N:M chatbot_knowledge_base
        ←  201 Chatbot

F — Ingesta cloud (pertenece a una KB)

Admin conecta credenciales OAuth de cloud (gdrive/s3) en /settings/integrations
Admin va a /knowledge/{kb}/documentos y "Añadir desde cloud"
  Cliente  →  POST /api/knowledge-bases/{kb}/sources/documents
              {cloud_folder_ref: {provider, folder_id, credential_id}}
          →  CloudStorageConnector.list_folder → registra DocumentSource por archivo
          ←  201 DocumentSource[]
  Cliente  →  POST /api/sources/{id}/ingest  (idem A; downloader → loader → chunker → ...)
Re-sync: manual desde panel (sin watcher en MVP).

G — Introspección de BD SQL (pertenece a una KB)

Cliente  →  POST /api/knowledge-bases/{kb}/sources/test-connection
            {spec: {driver, params, credential_id}}
        ←  200 {ok, error?}

Cliente  →  POST /api/knowledge-bases/{kb}/sources/databases
            {driver, credential_id, params}
        →  registra DatabaseSource
        ←  201 DatabaseSource

Cliente  →  POST /api/sources/{id}/introspect   (síncrono)
        →  SQLDataSource.introspect_schema → SchemaSnapshot
        →  persistir snapshot dentro de la entidad (sources.payload.schema_snapshot)
        ←  200 SchemaSnapshot

H — Ciclo agéntico de retrieval NUEVO

Streaming SSE: los endpoints de chat (panel y widget) usan text/event-stream. El cliente recibe eventos en vivo durante el loop para que el usuario vea el "pensamiento" del agente (qué tool está usando, en qué iteración va). La respuesta final viaja como un último evento del stream.

Tipos de evento SSE (catálogo cerrado):
  iteration_start { n }
  tool_call       { tool, args }
  tool_result     { summary, latency_ms }
  generating      { }
  final_answer    { message: { id, content, citations[] } }
  abstain         { reason }
  forced_close    { reason: 'max_iterations' }
  error           { code, message }
  done            { }

Invariantes:
  - El loop no puede exceder max_retrieval_iterations (1-5, default 3).
  - Si se alcanza el tope sin final_answer → forced_close + final_answer
    con lo acumulado; metadata.forced_close=true.
  - abstain corta el loop con un mensaje "no puedo responder eso".
  - Si agentic_mode=false, el router se invoca UNA sola vez y se ejecuta
    UNA sola tool antes de generar (pipeline lineal compatible).
  - El stream NUNCA emite tokens del LLM individualmente en MVP — solo
    el evento 'generating' y luego 'final_answer' con el mensaje completo.
    Esto da la sensación de "el agente está trabajando" sin requerir
    streaming token-a-token del LLM.

Puertos (contratos Python)

from abc import ABC, abstractmethod
from typing import AsyncIterator

class LLMProvider(ABC):
    @abstractmethod
    async def generate(self, prompt: str, config: GenerationConfig,
                       tools: list[ToolSpec] | None = None) -> LLMResponse: ...
    @abstractmethod
    async def stream(self, prompt: str, config: GenerationConfig) \
            -> AsyncIterator[str]: ...  # no usado en MVP
    @abstractmethod
    async def list_models(self) -> list[str]: ...
    supports_tool_calling: bool

class EmbeddingProvider(ABC):
    @abstractmethod
    async def embed_text(self, text: str) -> list[float]: ...
    @abstractmethod
    async def embed_batch(self, texts: list[str]) -> list[list[float]]: ...
    @abstractmethod
    def dimension(self, model_id: str) -> int: ...

class VectorStore(ABC):
    @abstractmethod
    async def ensure_collection(self, tenant_id: UUID, dim: int) -> None: ...
    @abstractmethod
    async def upsert(self, tenant_id: UUID, points: list[VectorPoint]) -> None: ...
    @abstractmethod
    async def search(self, tenant_id: UUID, kb_ids: list[UUID],
                     query_vec: list[float], top_k: int,
                     threshold: float | None) -> list[RetrievedChunk]: ...
    @abstractmethod
    async def delete_by_source(self, tenant_id: UUID, source_id: UUID) -> None: ...
    @abstractmethod
    async def delete_collection(self, tenant_id: UUID) -> None: ...

class Reranker(ABC):  # NUEVO
    @abstractmethod
    async def rerank(self, query: str, candidates: list[RetrievedChunk],
                     top_k: int) -> list[RetrievedChunk]: ...

class DocumentLoader(ABC):
    extensions: set[str]
    @abstractmethod
    async def load(self, uri: str) -> LoadedDocument: ...

class CloudStorageConnector(ABC):
    @abstractmethod
    async def list_folder(self, credential: ProviderCredential,
                          folder_ref: str) -> list[CloudFile]: ...
    @abstractmethod
    async def download(self, credential: ProviderCredential,
                       file_ref: str) -> bytes: ...
    @abstractmethod
    async def test_credential(self, credential: ProviderCredential) -> bool: ...

class SQLDataSource(ABC):
    driver: str
    @abstractmethod
    async def test_connection(self, spec: SQLConnectionSpec) -> bool: ...
    @abstractmethod
    async def introspect_schema(self, spec: SQLConnectionSpec) -> SchemaSnapshot: ...
    @abstractmethod
    async def execute_readonly(self, spec: SQLConnectionSpec, sql: str,
                               max_rows: int, timeout_s: float) -> SQLResult: ...

class Chunker(ABC):
    @abstractmethod
    def chunk(self, doc: LoadedDocument, config: ChunkingConfig) \
            -> list[Chunk]: ...

class Router(ABC):
    @abstractmethod
    async def route(self, state: ConversationState,
                    available_tools: list[ToolSpec]) -> ToolCall: ...

class QueryGuard(ABC):
    @abstractmethod
    async def validate(self, sql: str, schema: SchemaSnapshot,
                       judge_llm: LLMProvider | None) -> GuardVerdict: ...

class OAuthVerifier(ABC):
    @abstractmethod
    async def verify_token(self, id_token: str) -> OAuthProfile: ...

class SecretEncryptor(ABC):
    @abstractmethod
    def encrypt(self, plaintext: bytes) -> bytes: ...
    @abstractmethod
    def decrypt(self, ciphertext: bytes) -> bytes: ...

§6Casos de uso

34 use cases agrupados en 8 packages. Pivots clave aplicados desde la sesión 2: separación módulo LLM ↔ módulo RAG (las fuentes salen del chatbot a nivel tenant); KnowledgeBase como contenedor polimórfico; wizard cliente puro con un único POST final; RAG agéntico.

application/auth/

  • RegisterUser(email, password) → User
  • LoginUser(email, password) → JWT

  • BootstrapTenant(user_id) → Tenant — crea Tenant + colección Qdrant + ProviderCredential Ollama default (config_source=SERVER_ENV).

  • CreateChatbot(payload) → Chatbot — POST único; valida embedding compatibility cross-KB.
  • UpdateChatbot(chatbot_id, patch) → Chatbot

  • GetKnowledgeBase(kb_id) → KB + sources[]
  • DeleteKnowledgeBase(kb_id) → None — bloquea si algún chatbot referencia: KnowledgeBaseInUseError.

  • DetachSource(source_id) → None

  • Tenant scoping: automático vía middleware. Nunca se pasa tenant_id en URL ni body.
  • Content-Type: application/json (uploads: multipart/form-data).
  • Paginación: ?page=N&page_size=M (default 20, max 100); respuesta {items, page, page_size, total}.
  • IDs: UUIDv7.
  • Errores: {error: {code, message, details?}} + status HTTP. Códigos: 400 Validation, 401 Unauthenticated, 403 TenantScopeViolationError, 404 NotFound, 409 Conflict (incl. KnowledgeBaseInUse, IncompatibleEmbeddings), 422 Semantic, 429 RateLimited, 500 Internal.
  • Timestamps: ISO-8601 UTC.

Qdrant — colecciones

Una colección física por (tenant, dim): kb_chunks__<tenant_id>__<dim> (p. ej. kb_chunks__a1b2c3...__1024). Vector size = dim del embedding model. Distance: cosine. Las colecciones se crean on-demand al crear la primera KB con un dim nuevo en el tenant.

Las KBs adjuntas a un mismo chatbot siguen obligadas a compartir embedding_selection (regla validada upstream con IncompatibleEmbeddingsError). Esto garantiza que el search de un chatbot vive en una sola colección por dim y los kb_ids permitidos están todos en ella.

Payload por punto:

{
  tenant_id: uuid,      (filtro mandatorio en TODO query — defensa en
                         profundidad pese a tener colección por tenant)
  kb_id: uuid,          (filtro por kb_id en search)
  source_id: uuid,
  chunk_index: int,
  content: string,      (texto del chunk — única copia)
  metadata: { page_number?, section?, source_filename?, ... }
}

Filtro default en search:

// Resolver colección: dim = chatbot.kb_ids[0].embedding_selection.dim
collection = f"kb_chunks__{ctx.tenant_id}__{dim}"
must = [
  { key: 'tenant_id', value: ctx.tenant_id },
  { key: 'kb_id', values: chatbot.kb_ids }
]

Borrado de tenant: al borrar un tenant se enumeran todas las colecciones que matchean el prefijo kb_chunks__<tenant_id>__* y se borran en cascada.

Invariantes de integridad

  • Todo SELECT/UPDATE filtra por tenant_id (vía repository pattern; los repos solo aceptan ctx con tenant_id validado).
  • sources.type='database' ⇒ payload->>'driver' IN ('postgres','mysql').
  • sources.type='document' ⇒ payload->>'kind' IN ('upload','cloud').
  • Borrar una KB con FK RESTRICT en chatbot_knowledge_base.kb_id falla → se traduce a KnowledgeBaseInUseError.

Alembic. Una migración por feature/proposal. Reversibles. Cero migraciones manuales en datos productivos.

Ciclo de vida del bundle

  1. Lee data-chatbot-id.
  2. GET /api/public/chatbots/{id}/widget-config → recibe colores, fuente, posición, avatar, greeting, allowed_domains.
  3. Comprueba window.location.origin contra allowed_domains. Si no coincide, no monta nada y emite warning a console.
  4. Inyecta container con Shadow DOM cerrado y monta UI (Preact). Aísla CSS host del widget.
  5. Sesión: lee/escribe cookie tfm_widget_session_<chatbot_id> (HttpOnly first-party desde el backend en primera respuesta).
  6. UI: floating button → click abre panel (resizable en desktop, fullscreen en móvil < 640px).

Envío de mensajes

POST /api/public/chatbots/{id}/chat con {session_id?, message}. El servidor responde con SSE stream (text/event-stream). El widget consume los eventos del loop agéntico (ver §3-H) y muestra una "burbuja activa" con el estado:

  • iteration_start → cambia copy "Pensando (paso N)…"
  • tool_call: search_docs → "Buscando en documentos…"
  • tool_call: query_database → "Consultando base de datos…"
  • tool_result → "Procesando resultados…"
  • generating → "Escribiendo respuesta…"
  • final_answer → materializa la burbuja del bot con el mensaje + citas
  • abstain → mensaje cordial de "no puedo responder eso"
  • error → "Algo salió mal, inténtalo de nuevo" + retry button

Si la conexión SSE se corta, el widget reintenta automáticamente la próxima vez que el usuario envíe un mensaje (no resume el stream interrumpido).

§12Seguridad y multi-tenancy

Modelo de amenazas

Activos: datos del tenant (documentos, snapshots de BD, conversaciones); credenciales de proveedores (cifradas); credenciales de BDs SQL del cliente; JWTs de sesión.

Atacantes considerados: usuario autenticado que intenta cruzar tenants; visitante anónimo del widget público; atacante con XSS en la web del cliente final que aloja el widget; operador interno con acceso a BD.

Defensa en profundidad (7 capas)

Capa 1 — Network / Edge

  • HTTPS recomendado en cualquier deploy expuesto. En el MVP local del TFM (docker-compose) el stack corre en HTTP plano; la dependencia de HTTPS queda como nota para producción.
  • CORS estricto: panel solo desde origen del frontend Next.js; widget público desde allowed_domains del chatbot.
  • Rate limiting: IP-based en login/register, chatbot-based en chat público (ver §8).

Capa 2 — AuthN

  • JWT firmado HS256 con secreto rotable (JWT_SECRET en .env). Migración a RS256 planeada post-MVP.
  • Expiración 24h. Sin refresh tokens en MVP — el cliente re-login al expirar. NextAuth gestiona la sesión cliente y refresca la cookie cuando el backend emite un nuevo JWT.
  • Google OAuth vía NextAuth + google-auth en backend para re-verificar el id_token antes de emitir el JWT propio.

Capa 3 — AuthZ + tenancy

  • TenantScopingMiddleware extrae tenant_id del JWT y lo inyecta en RequestContext.
  • TODOS los repositorios reciben ctx y filtran por ctx.tenant_id; nunca aceptan tenant_id del request body.
  • Cualquier intento de acceso a recurso con tenant distinto → TenantScopeViolationError (403).
  • Tests de mutación obligatorios por endpoint que intentan tenant_B leyendo datos de tenant_A.
  • Qdrant: colección física por (tenant, dim) (no solo filtro lógico). Filtros adicionales por kb_id y tenant_id en payload.
  • Storage: prefix tenant_<id>/... por convención + ACL si se usa S3.

Capa 4 — Secrets at rest

  • Fernet (AES-128-GCM + HMAC) con clave maestra de 32 bytes en env var FERNET_KEY.
  • Cifrados: provider_credentials.api_key, provider_credentials.base_url (opcional), source_db_credentials.connection_string.
  • Rotación de clave maestra: post-MVP (procedimiento documentado).

Capa 5 — SQL execution safety

Triple guard en ExecuteSQL:

  1. sqlglot AST: solo SELECT, sin cross-DB JOIN, sin acceso a pg_catalog/information_schema (configurable por driver).
  2. LLM semántico: revisa prompt + SQL generado contra patrones sensibles (SSN, passwords, salarios, …). Modelo barato (router_llm_selection si está definido).
  3. Límites duros: max_rows=500, timeout=10s, max_cost cuando soportado por driver.

Conexión SQL: usuario read-only obligatorio, validado en AttachDatabaseSource intentando un INSERT que debe fallar.

Capa 6 — Input validation

  • Pydantic en cada request body.
  • Tamaño máximo de upload: 50 MB por archivo, 10 archivos por multipart request.
  • System prompt: max 4 KB.

Capa 7 — Output filtering

  • Markdown rendering en panel con sanitización (rehype-sanitize) para evitar XSS en citas.
  • Widget renderiza texto plano por defecto; markdown opt-in con el mismo sanitizer.
  • CSP headers estrictos en el panel.

Logging y audit

  • structlog en JSON. Campos: timestamp, level, tenant_id, user_id, chatbot_id?, action, latency_ms, status.
  • PII redaction en logs (email, api_keys) por filtros del logger.
  • Retención: 30 días en disco (rotación), 1 año en archivo comprimido (planificado).

Dependencias de seguridad críticas

bcrypt (cost 12), cryptography (Fernet), python-jose o pyjwt, google-auth, sqlglot.

§13Evaluación (RAGAS)

Objetivo

Comparar variantes del sistema (con/sin reranker, distintos chunk sizes, distintos LLMs) sobre datasets controlados para sustentar las afirmaciones cuantitativas de la memoria del TFM.

Framework

RAGAS. Métricas en MVP:

MétricaQué mide
faithfulness¿La respuesta solo afirma lo que aparece en el contexto recuperado?
answer_relevance¿La respuesta aborda la pregunta?
context_precision¿El contexto recuperado es pertinente?
context_recall¿Se recuperó el contexto necesario para responder?

LLM-as-judge: usa el modelo del chatbot evaluado por defecto, o un override --judge-model.

Dataset

Estrategia: primer intento es reutilizar un dataset público que encaje (e.g. RAG-12000, MS MARCO QA, FiQA para finanzas, o un subset filtrado de SQuAD). Si ninguno encaja, se construye uno propio en M6 a partir de los PDFs académicos y la BD demo (chinook).

Formato JSONL, una línea por ejemplo:

{ "question": "...",
  "ground_truth": "...",
  "scenario": "doc_only" | "sql_only" | "mixed" | "abstain",
  "metadata": { "difficulty": "easy"|"medium"|"hard",
                "source_dataset": "public:rag-12000" | "tfm:custom" } }

La preferencia es dataset público para reducir trabajo manual; el campo source_dataset permite mezclar ambos en el mismo experimento si hace falta.

Pipeline de ejecución

  1. CLI lee dataset_path.
  2. Para cada question: POST /api/chatbots/{id}/chat con sesión efímera (flag --no-persist que no escribe en chat_sessions).
  3. Recolecta {question, answer, contexts, ground_truth}.
  4. RAGAS computa métricas en batch.
  5. Output:
    • report.json con métricas agregadas y por ejemplo.
    • report.md con tabla resumen + top-3 fallos por métrica.

Escenarios del TFM

ScenarioConfiguraciónPreguntas
doc_onlyKB con ~20 PDFs académicosSobre contenido textual
sql_onlyKB con BD demo (chinook)Analíticas
mixedKB con PDFs + chinookRequieren ambos

Variables a comparar (matriz de evaluación)

  • reranker on/off
  • agentic_mode true/false
  • max_retrieval_iterations ∈ {1, 3, 5}
  • chunk_size ∈ {256, 512, 1024}
  • LLM ∈ {ollama:llama3, openai:gpt-4o-mini}

Reproducibilidad

Seed fijo (temperatura 0 en LLM-as-judge). Versiones de modelo grabadas en el report. Dockerfile.eval con dependencias congeladas.

§15Riesgos y mitigaciones

IDRiesgoProb.ImpactoMitigación
R-01Filtración cross-tenantBajaCrítico Tests de mutación obligatorios por endpoint. Repository pattern estricto. Colecciones Qdrant físicas por tenant. Code review bloqueante en endpoints nuevos.
R-02SQL injection vía text2SQLMediaCrítico 3 capas de guards (AST + LLM semántico + límites). Usuario read-only obligatorio validado en attach. Tests adversariales en eval.
R-03Coste runaway por loop agéntico mal calibradoMediaAlto max_retrieval_iterations ≤ 5. final_answer priorizada en system prompt. router_llm_selection (modelo barato) opt-in. Monitoring de iteraciones medias en eval (alerta > 2).
R-04Embedding mismatch entre KBs adjuntas a un chatbotAltaAlto Validación dura en CreateChatbot (IncompatibleEmbeddingsError). UI filtra opciones tras la 1ª selección. Tests de integración para el path de error.
R-05PDFs/docs malformados rompen ingestaAltaMedio Cada loader maneja sus excepciones y marca el source como failed con error legible. Frontend muestra error y permite reintento.
R-06Sesión JWT no se revoca en logoutAltaBajo Por diseño en MVP. TTL 24h como compromiso. Documentado como limitación conocida. Roadmap post-MVP: blacklist en Redis o refresh tokens.
R-07Calidad de retrieval insuficiente para SQL complejosMediaAlto Non-goal explícito: no reemplazar a un analista. Sample rows en schema snapshot. Eval scenario sql_only con baseline conservador.
R-08Cuello de botella en Qdrant con muchos tenantsBajaAlto Colecciones físicas por tenant facilitan sharding futuro. Monitor de tamaño por colección. Documentado para re-evaluar en producción.
R-09Dependencia de modelos cloud puede caer / cambiar APIMediaMedio Adapter pattern aísla en un archivo por proveedor. Catálogo en código permite añadir nuevos rápido. Ollama como fallback siempre disponible.
R-10Ataque XSS desde citas en el widget públicoBajaMedio Shadow DOM cerrado en widget. rehype-sanitize en markdown rendering. CSP headers estrictos. Tests de regresión.
R-11Confusión usuario por wizard sin draft + indexación previaAltaMedio Empty state en wizard sugiere crear KB primero si no hay ninguna. Mensaje claro: "Las KBs se crean en /knowledge".
R-12RAGAS LLM-as-judge da scores inestablesMediaMedio Temperatura 0 obligatoria. Múltiples runs y mediana. Documentar limitaciones de la métrica en la memoria del TFM.
